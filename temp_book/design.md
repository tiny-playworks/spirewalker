# 项目设计文档（差异化落地版 v0.2）

本文档不是单纯的玩法脑暴稿，而是服务当前项目实现的产品设计稿。

目标很明确：

- 参考成熟的单人爬塔卡牌 Roguelike 结构经验，但不依赖任何单一现成作品作为模板
- 优先建立自己的核心机制、内容语义和系统边界，而不是沿用既有经典套路
- 在没有成熟美术资源的前提下，先把可玩性、可读性、可扩展性做扎实
- 文档要贴合当前工程，而不是另起一套空中楼阁

---

## 1. 项目定位

### 1.1 一句话定位

这是一个以“节奏、连段、蓄势爆发”为核心体验的单人爬塔卡牌 Roguelike 原型。

它采用的成熟结构包括：

- 回合制战斗
- 地图分支推进
- 明牌敌方意图
- 战后成长与构筑变化

它刻意避免的是：

- 直接套用现成作品的角色模板和职业结构
- 依赖固定流派标签来组织内容和成长路线
- 仅靠数值近似和熟悉套路来堆砌内容
- 在题材、美术和系统表达上形成过强既视感

### 1.2 当前产品阶段

当前项目更接近“可玩的系统原型”而不是“内容完整的商业游戏”。

现阶段优先级：

1. 核心循环成立
2. 差异化机制成立
3. 系统扩展点稳定
4. 低成本表现可接受
5. 大量内容和高规格美术靠后

---

## 2. 差异化方向

如果只是做一个“高度贴近成熟前作体验”的项目，它最后一定会输在内容量、美术识别度和完成度上。

所以本项目必须从第一性原理出发，明确自己真正要做什么。

这类爬塔卡牌游戏的关键，不是“卡牌长什么样”，而是：

- 每回合资源如何分配
- 已知风险下如何做取舍
- 构筑如何逐步成型
- 玩家如何在少量信息中感受到掌控感

本项目的差异化，不靠表面题材换皮，主要靠战斗节奏重心变化。

### 2.1 核心差异化主轴：势

本项目建议把“势”作为第一核心系统。

“势”不是单独一个数值 UI，而是一组围绕节奏展开的设计原则：

- 玩家可以通过连续出牌、特定卡牌、状态或遗物累积优势
- 优势不只是即时伤害，也可以是下回合的主动权、减伤、额外收益或爆发窗口
- 爆发不是无代价的，通常伴随衰减、失衡、手牌压力或防御空窗

这比单纯做“力量 / 敏捷 / 中毒”的常规组合更适合做出辨识度。

### 2.2 玩家体验目标

理想体验不是“每张卡都认识”，而是“这一回合我要不要继续压节奏”。

玩家每回合都应该面对类似判断：

- 我要现在把势拉满，还是先稳住
- 我这一轮爆发后，下轮会不会断档
- 我是为了立刻击杀，还是为了下回合建立更大优势
- 这套牌是追求持续压制，还是抓一个关键爆发点

这会让玩法从“普通攻防交换”转向“节奏管理”。

### 2.3 现阶段最合适的差异化落点

结合当前代码状态，最合适的切入点不是一次性发明很多复杂系统，而是先把以下方向做强：

- 连续出牌收益
- 回合内资源转化
- 自动消耗与敌人打断
- 明确的爆发窗口
- 敌人对玩家节奏的干扰

你现有代码里已经有 `momentum` 这种雏形，这个方向是顺的，成本也低。

## 3. 核心玩法循环

### 3.1 Run 外循环

一局游戏的基础循环仍然采用成熟的爬塔框架：

1. 地图选路
2. 进入战斗 / 事件 / 商店 / 休息点 / 宝箱
3. 获得卡牌、遗物、药水或资源变化
4. 构筑逐渐成型
5. 挑战更强敌人与 Boss

原因很简单：这套外循环已经证明有效，没必要为了“创新”而放弃成熟骨架。

真正需要做差异化的是战斗内部。

### 3.2 战斗内循环

战斗每回合的基础节奏：

1. 观察敌方意图
2. 判断本回合是“蓄势”、“过渡”还是“爆发”
3. 出牌并塑造下一回合资源结构
4. 承受敌方回合压力
5. 在持续推进中寻找一轮决定性压制

这里的重点不是“多打一张牌”，而是“怎样把一回合的收益转成两回合的优势”。

## 4. 战斗设计核心

### 4.1 基础资源

基础资源仍保持简单：

- 能量
- 手牌
- 抽牌堆 / 弃牌堆 / 消耗堆
- 生命
- 格挡
- 状态

这些都是当前项目已经具备或已经在模型中预留的部分，没必要再造新轮子。

### 4.2 第一批核心机制建议

MVP 阶段建议聚焦三类机制：

#### 一类：蓄势

作用：让玩家为了未来回合提前投资。

典型设计方向：

- 打出后获得若干层“势”
- 每次出牌触发额外格挡、伤害或抽牌
- 在本回合不立刻赚很多，但会改变后续决策

#### 二类：爆发

作用：把前几回合积累转换成一轮高收益行动。

典型设计方向：

- 消耗势以提高伤害或连击次数
- 根据本回合已出牌数强化效果
- 将格挡、状态或弃牌转化为攻击收益

#### 三类：失衡

作用：限制无脑堆爆发，制造节奏波动。

典型设计方向：

- 爆发后下回合减益
- 手牌效率下降
- 抽牌/费用受限
- 势被转化或被敌人打断

没有“失衡”，爆发系统最后只会变成另一个数值膨胀按钮。

### 4.3 为什么这样设计

从第一性原理看，回合制卡牌战斗想成立，必须同时满足两件事：

- 玩家能建立优势
- 优势不能无限无脑滚大

“蓄势 -> 爆发 -> 失衡恢复”正好构成一个自然闭环。

### 4.4 势（momentum）最小规则定义

本节不是概念描述，而是 v0.3 可直接指导实现的最小规格。

```ts
type MomentumSpec = {
  statusId: "momentum";
  representation: {
    kind: "stack";
    owner: "player_unit";
    min: 0;
    max: 8;
    resetOnBattleEnd: true;
  };
  gain: [
    {
      source: "card_effect";
      effectType: "apply_status";
      target: "self";
      amount: "1..3";
    },
    {
      source: "relic_or_event";
      effectType: "apply_status";
      target: "self";
      amount: "1..2";
    },
  ];
  consume: [
    {
      mode: "automatic";
      trigger: "onAfterPlayCard";
      excludeCardIds: ["momentum"];
      amount: 1;
      payoff: {
        type: "block";
        value: "equal_to_current_stacks";
      };
    },
    {
      mode: "active";
      trigger: "specific_burst_card";
      amount: "all | fixed";
      payoff: {
        type: "damage";
        value: "card_defined";
      };
    },
  ];
  decay: {
    turnEnd: "none";
    noTriggerThisTurn: "none";
    interruptedByEnemy: ["reduce_stacks", "clear_stacks", "punish_multi_play"];
  };
  affects: {
    damage: "indirect_only";
    draw: "indirect_only";
    energy: "none_in_v0_3";
    block: "direct_by_status_hook";
  };
  enemyInteraction: {
    canInterfere: true;
    canCounter: true;
    methods: ["reduce_stacks", "clear_stacks", "punish_multi_play"];
  };
};
```

结算顺序约束：

- 卡牌主效果必须先结算完成，再触发 `onAfterPlayCard` 对 `momentum` 的自动消耗。
- 如果某张卡本身既能获得 `momentum`，又会触发自动消耗，则该次自动消耗读取的是“卡牌主效果结算后”的当前层数。
- 主动消耗 `momentum` 的爆发卡，其消耗与收益结算优先于该次出牌后的自动消耗，避免一次出牌同时被双重扣减。

补充约束：

- `momentum` 在本阶段本质上是 `status stack`，不是独立 meter，不单独挂在 `BattleState` 顶层。
- 基础 `momentum` 不直接改写基础伤害、基础抽牌、基础能量。
- 本阶段只允许它直接影响格挡；对伤害和抽牌的影响只能通过特定卡牌“转化”实现。
- 本阶段不允许 `momentum` 直接产能，否则节奏会很快失控。
- 敌人必须能干扰 `momentum`，但不要求所有敌人都能干扰。
- v0.3 至少要有 1 个普通敌人和 1 个精英敌人具备反制 `momentum` 的手段。

### 4.5 势的策略角色

`momentum` 不只是一个数值系统，它在战斗中的真正作用是定义玩家当前所处的节奏位置。

在策略层，玩家每个回合通常会处于以下三类状态之一：

- 蓄势回合：优先积累 `momentum`，允许牺牲部分即时收益，以换取后续更强的转化空间
- 爆发回合：主动将 `momentum` 转化为高收益动作，追求在单回合内建立明显优势
- 过渡回合：在 `momentum` 较低、被敌人打断或爆发后失衡时，优先恢复手牌效率、防御能力或下一轮节奏

这一层设计的目标不是单纯让玩家“多一个可叠层状态”，而是让以下判断持续成立：

- 同一手牌在不同 `momentum` 层数下，最优出牌顺序会发生变化
- 玩家需要在“继续蓄势”和“立即兑现”之间反复权衡
- 敌人的干扰能力会实质性改变玩家对当回合节奏的判断

如果 `momentum` 只提供线性数值收益，而不能改变回合决策顺序，那么它就退化成了普通 buff，而不是本项目的核心差异化系统。

## 5. 当前工程对齐

本设计必须服从当前工程结构。

当前项目已经形成了比较健康的分层：

- `React` 负责页面和交互
- `Zustand Store` 负责状态承接与 selector
- `GameEngine` 负责命令分发和规则变更
- `core/model` 负责稳定数据结构
- `core/systems` 负责战斗、地图、奖励、商店、事件等规则模块
- `Phaser` 负责表现层和交互桥接

这套边界是对的，不应该推翻。

### 5.1 文档必须对齐的真实模型

当前运行主状态已经比较清晰：

- `RunState` 管总流程
- `ScreenState` 负责页面流转
- `BattleState` 负责战斗状态机
- `MapState` 负责地图推进
- `RewardState / ShopState` 负责阶段性流程

真实代码可参考：

- `src/game/core/model/run.ts`
- `src/game/core/model/battle.ts`
- `src/game/core/model/map.ts`

### 5.2 设计落地的第一原则

以后新增玩法时，优先沿用现有入口：

- 新交互先看是否只是新增 `GameCommand`
- 新表现先看是否只是新增 `GameEvent`
- 新状态优先走 `status` 注册和 hook
- 新敌人优先走 `enemyAi` 抽象
- 新卡牌优先走 `CardDefinition.effects`

不要把设计写成“将来可能有个超级系统”，最后却绕过当前引擎结构。

### 5.3 当前最值得继续强化的扩展点

从已有实现看，以下点最适合继续做深：

- `EffectDefinition`
- `status hook`
- `enemy intent`
- `reward / shop / map` 的节奏分配

参考文件：

- `src/game/core/model/card.ts`
- `src/game/core/model/status.ts`
- `src/game/core/systems/enemy/enemyAi.ts`

## 6. 内容设计原则

### 6.1 少而清晰，不靠堆量

前期内容设计不追求数量，追求可辨识。

坏内容扩充方式：

- 先做几十张数值换皮卡
- 先抄一套常见状态体系
- 先铺很多敌人但行为没区分

好内容扩充方式：

- 每张卡都有明确战术角色
- 每种状态都服务某种节奏判断
- 每个敌人都在破坏玩家某类节奏

### 6.2 卡牌设计原则

每张卡至少回答一个问题：

- 它是帮助蓄势，还是帮助爆发
- 它是补稳定性，还是换上限
- 它在什么回合最强
- 它和哪类状态、遗物、敌人行为有互动

如果一张卡只是“另一个数值版本的打击/防御”，那它不该优先做。

### 6.3 状态设计原则

状态不宜太多，先把少量状态做清楚。

建议优先保留三类：

- 直接战斗修正：如力量、虚弱、易伤
- 节奏状态：如连势、充能、迟滞
- 敌我节奏干扰状态：如限制爆发、惩罚连打、打断蓄势

当前代码已实现或已预留的状态很适合做第一批样板：

- `strength`
- `weak`
- `vulnerable`
- `momentum`
- `metallicize`

参考：

- `src/game/core/definitions/statuses.ts`

### 6.4 敌人设计原则

敌人的价值，不是充当血包，而是逼玩家调整节奏。

第一批敌人就应该分工明确：

- 有的敌人逼你抢伤害
- 有的敌人逼你留防守
- 有的敌人专门克制连打
- 有的敌人拖长战斗，逼你证明构筑上限

当前敌人 AI 还比较早期，主要是交替攻击，这没问题，说明工程骨架先立起来了。

下一步应优先做“行为差异”，不是先做“更复杂数值”。

参考：

- `src/game/core/definitions/monsters/definitions.ts`
- `tests/monster/enemyAi.test.ts`

## 7. 低美术资源下的表现策略

目前没有成熟美术资源，这不是问题，前提是设计别依赖高成本表现。

### 7.1 当前阶段的正确目标

不是“做得像正式商业成品”，而是“让玩家一眼看懂局面，并愿意继续打下一回合”。

### 7.2 表现优先级

前期应优先保证：

1. 信息可读
2. 反馈明确
3. 操作流畅
4. 主题统一
5. 高规格美术最后再补

### 7.3 建议的低成本视觉方向

建议采用抽象化、符号化、界面驱动的表现方式：

- 角色和敌人可以先用剪影、图形块、简化立绘占位
- 通过颜色、轮廓、姿态差异区分敌人角色
- 强调意图图标、状态图标、伤害数字、战斗日志
- 让牌面、资源区、敌方意图区承担主要信息表达

这样做的好处是：

- 不依赖成套角色原画
- 不依赖复杂骨骼动画
- 先把玩法读感做出来
- 后续替换美术时不需要推翻规则层

### 7.4 风格建议

既然目标不是复用某个成熟前作的视觉印象，就不要做“纸片地牢手绘感”的近似风格。

更适合当前阶段的方向是：

- 冷静
- 抽象
- 轻压迫感
- 少量高对比强调色
- UI 统一强于角色细节

## 8. MVP 范围

### 8.1 MVP 目标

MVP 不是“内容足够多”，而是“闭环成立且有一条清楚的差异化体验”。

### 8.2 建议 MVP 内容范围

- 1 个可玩的角色原型
- 1 套清晰的核心节奏机制
- 8 到 15 张高辨识度卡牌
- 3 到 5 个功能明确的普通敌人
- 1 到 2 个精英敌人
- 1 个 Boss 原型
- 1 层到 2 层地图
- 基础事件、商店、休息点、奖励闭环

这个规模足够验证方向，不会把开发拖进内容泥潭。

### 8.3 MVP 不做什么

前期明确不做：

- 多职业并行开发
- 大量稀有卡和复杂遗物池
- 多章节世界观叙事
- 重美术驱动的角色演出
- 复杂剧情事件树
- 过早的数值精细平衡

## 9. 版本推进建议

### 9.1 v0.2 到 v0.3

目标：把“节奏差异化”从概念变成可玩反馈。

建议优先事项：

- 明确 `momentum` 的最小规则，并把它固定成状态驱动方案
- 做出 1 批围绕蓄势 / 爆发 / 失衡的卡牌
- 增加 2 到 3 类真正不同的敌人行为
- 让奖励、商店、事件能影响这套节奏体系

### 9.2 v0.3 到 v0.4

目标：让构筑开始真正分化。

建议方向：

- 遗物开始强化不同节奏路线
- 事件开始提供风险收益型选择
- 地图节点分布服务不同构筑决策
- Boss 负责检测玩家是否真的成型

### 9.3 v0.4 之后

等验证出真正好玩的核心后，再谈：

- 内容扩容
- 视觉升级
- 音效强化
- 数值打磨
- 世界观包装

顺序不能反。

## 10. 版本验收标准

以下标准都必须可验证，不用“感觉不错”这种说法。

### 10.1 势系统闭环成立

- 至少 3 张卡可以获得 `momentum`
- 至少 2 张卡会主动消耗 `momentum`
- 至少 1 个敌人会减少或清空 `momentum`
- 在测试局中，必须能实际触发“获得 -> 自动消耗 -> 敌方干扰”这三个环节

### 10.2 节奏差异可验证

- 至少 3 张牌的收益明显依赖当前 `momentum` 层数或是否存在 `momentum`
- 至少存在 2 类不同回合决策：
  - 先蓄势再爆发
  - 先稳住再等待下轮
- 同一套牌在不同 `momentum` 层数下，最优出牌顺序应发生变化

### 10.3 构筑分化成立

- 至少存在 2 条可描述的构筑路径
- 每条路径至少有 4 张支持卡牌
- 每条路径至少有 1 个专属强化方向的遗物
- 两条路径都能通过首个精英战验证，不要求数值完全平衡

### 10.4 敌方反制成立

- 至少 1 个普通敌人具备节奏干扰能力
- 至少 1 个精英敌人具备直接针对 `momentum` 的反制能力
- 至少 1 个敌人会迫使玩家减少连打或改变出牌顺序

### 10.5 工程边界不被破坏

- 基础 `momentum` 循环不新增 `GameCommand`
- `GameEngine.dispatch` 不为 `momentum` 单独增加分支
- `momentum` 的基础流转应主要通过 `status`、`effect`、已有战斗结算 hook 完成

---

## 系统设计约束

为保证 v0.3 阶段的可扩展性与工程稳定性，以下约束必须成立：

- `momentum` 是本阶段唯一新增的核心节奏资源，不引入第二套平行核心资源系统
- `momentum` 必须继续通过 `status + effect + hook` 实现，不建立专用执行通道
- `GameEngine.dispatch` 不因 `momentum` 增加专门分支逻辑
- 所有基础 `momentum` 行为都应能够通过已有事件语义表达，用于表现层反馈与调试
- `momentum` 不直接改写基础攻击、基础抽牌和基础能量规则，只通过状态钩子或特定卡牌转化影响结果

出现以下情况时，应视为结构风险，并重新评估方案是否偏离当前设计：

- 为了少量卡牌效果而新增一套只服务 `momentum` 的底层机制
- 为了表现层需求反向修改规则层边界
- 为了增强“爽感”而让 `momentum` 直接产能或直接线性放大所有收益

## 工程附录 A：势系统落地

### A.1 对应模块

`momentum` 在本阶段的落点应固定为：

- `status`
  - 使用现有 `momentum` 状态 id
  - 继续通过 `status` 注册表挂 `onAfterPlayCard` 行为
- `effect`
  - 获取势：复用现有 `apply_status`
  - 消耗势做爆发：优先走现有 `custom`，先不要新增 effect 类型
- `GameEvent`
  - 获取势：先复用 `STATUS_APPLIED`
  - 自动转格挡：先复用 `BLOCK_GAINED`
  - 本阶段不新增 `MOMENTUM_CHANGED`

### A.2 最小实现路径

最小实现路径不是重做引擎，而是在现有结构上补齐规则。

1. 保留 `momentum` 作为 `status stack`
2. 用 `apply_status` 让卡牌、遗物、事件可以获得 `momentum`
3. 保留 `onAfterPlayCard` 的自动消耗逻辑，作为基础收益层
4. 新增 2 张到 3 张“爆发卡”，通过 `custom` 把 `momentum` 转成伤害或抽牌
5. 给 1 个普通敌人和 1 个精英敌人增加“降势/清势/惩罚连打”能力
6. 为 `momentum` 补充规则测试，而不是改 `dispatch` 结构

### A.3 是否需要改 GameEngine

- 基础 `momentum` 循环：不需要改 `GameEngine`
- 新增蓄势卡：不需要改 `GameEngine`
- 新增爆发卡：优先复用现有 `custom effect` 入口，不改 `GameEngine.dispatch`
- 新增敌人干扰势：优先落在敌方意图和战斗结算逻辑，不改命令分发结构

结论：

- 本阶段不是“新增一整套势系统框架”
- 本阶段是“沿用现有 status/effect/hook，把 momentum 做实”

### A.4 本阶段绝对不做

- 不把 `momentum` 提升成 `BattleState` 顶层专属 meter
- 不新增一组只服务 `momentum` 的命令类型
- 不新增一组只服务 `momentum` 的独立事件体系
- 不做 `momentum` 直接产能
- 不做超过 1 个以上的势变体状态
- 不为了爆发卡少量需求就扩一个新的通用 effect 大系统

## 11. 设计红线

后续所有设计都尽量不要踩这些坑：

- 不要因为“贴近成熟前作更安全”而直接复用经典卡牌套路
- 不要在差异化还没立住前就扩卡池
- 不要让表现层反向绑架规则设计
- 不要在没有美术资源时设计强依赖演出的系统
- 不要为了“复杂”而堆系统名词
- 不要让文档脱离当前代码结构

---

## 12. 当前结论

这个项目最合理的路，不是去复刻某个成熟前作，而是借鉴这一品类已经验证过的外循环骨架，做一个更强调“节奏管理、连段收益、蓄势爆发”的爬塔卡牌游戏。

短期内最重要的不是做更多内容，而是把这三个点做实：

- 差异化核心是否成立
- 当前引擎结构是否能稳定承载扩展
- 在低美术条件下，战斗是否依然清楚且好玩

如果这三点成立，后面的内容扩展才有意义。

## 13. 常见失败模式（设计预警）

在后续实现过程中，需要持续警惕以下失败模式：

### 13.1 `momentum` 退化为普通数值增益

表现：

- 主要用途只剩下提高伤害或格挡数值
- 玩家不需要因为 `momentum` 改变出牌顺序

结果：

- 系统失去节奏差异化，退化为常规 buff

### 13.2 爆发没有真实成本

表现：

- `momentum` 获取过于稳定，且收益过高
- 爆发后没有明显失衡、断档或防御空窗

结果：

- 最优策略会收敛为无脑堆叠并立即兑现

### 13.3 敌人无法实质性干扰节奏

表现：

- 敌人只做数值攻击或防御
- 敌方行为不会影响 `momentum` 或玩家出牌顺序

结果：

- 战斗会退化为单向滚雪球，节奏系统缺少外部压力测试

### 13.4 表现层反向绑架规则设计

表现：

- 为了动画、视觉演出或日志展示去改写底层规则
- 引入只服务表现层的特殊状态或特殊结算

结果：

- 规则复杂度上升，系统边界变脏，后续扩展成本显著增加
